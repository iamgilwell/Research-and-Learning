# Kubernetes Glossary - Complete Reference

## A

**Admission Controller**
- Plugin that intercepts API requests before objects are persisted
- Can validate, mutate, or reject requests
- Examples: ResourceQuota, PodSecurityPolicy, NetworkPolicy

**Affinity**
- Rules that influence pod scheduling decisions
- **Node Affinity**: Attract pods to specific nodes
- **Pod Affinity**: Co-locate pods together
- **Anti-Affinity**: Separate pods from each other

**Annotation**
- Key-value metadata attached to objects
- Not used for selection, only for informational purposes
- Example: `deployment.kubernetes.io/revision: "3"`

**API Group**
- Collection of related API resources
- Examples: `apps/v1`, `networking.k8s.io/v1`
- Allows versioning and organization of APIs

**API Server**
- Central management component of Kubernetes
- Exposes REST API for all operations
- Validates and processes all cluster requests

## B

**Binary**
- Executable file for Kubernetes components
- Examples: kubectl, kubelet, kube-apiserver
- Distributed as part of Kubernetes releases

**Bootstrap Token**
- Temporary token used for node joining
- Automatically expires after set time
- Used during cluster initialization

## C

**CGroup (Control Group)**
- Linux kernel feature for resource isolation
- Controls CPU, memory, I/O usage
- Used by container runtimes for resource limits

**CIDR (Classless Inter-Domain Routing)**
- Method for IP address allocation
- Example: `10.244.0.0/16` for pod network
- Used in network configuration

**Cluster**
- Set of nodes running Kubernetes
- Includes control plane and worker nodes
- Provides unified compute, network, and storage

**ClusterIP**
- Default service type for internal communication
- Only accessible within cluster
- Provides stable IP for service discovery

**ConfigMap**
- API object for storing non-confidential configuration
- Key-value pairs or file-like data
- Can be mounted as volumes or environment variables

**Container**
- Lightweight, portable execution environment
- Packages application with dependencies
- Runs on container runtime (Docker, containerd)

**Container Runtime Interface (CRI)**
- API for container runtime integration
- Allows pluggable container runtimes
- Examples: Docker, containerd, CRI-O

**Controller**
- Control loop that watches cluster state
- Takes action to move current state toward desired state
- Examples: Deployment Controller, ReplicaSet Controller

**CronJob**
- Kubernetes object for scheduled jobs
- Uses cron syntax for scheduling
- Creates Jobs at specified times

**Custom Resource (CR)**
- Extension of Kubernetes API
- Defined by Custom Resource Definition (CRD)
- Allows application-specific objects

**Custom Resource Definition (CRD)**
- Schema for custom resources
- Extends Kubernetes API with new object types
- Used by operators and advanced applications

## D

**DaemonSet**
- Ensures pod runs on all (or some) nodes
- Automatically schedules on new nodes
- Used for system services (logging, monitoring)

**Deployment**
- Manages ReplicaSets and provides declarative updates
- Supports rolling updates and rollbacks
- Most common way to run stateless applications

**Docker**
- Container platform and runtime
- Creates and manages container images
- Can be used as Kubernetes container runtime

**Downward API**
- Mechanism to expose pod/container metadata
- Available as environment variables or files
- Examples: pod name, namespace, node name

## E

**Endpoint**
- Network address (IP:port) of a pod
- Automatically managed by Services
- Used for load balancing traffic

**EndpointSlice**
- Scalable alternative to Endpoints
- Supports larger clusters more efficiently
- Automatically created by Services

**etcd**
- Distributed key-value store
- Stores all Kubernetes cluster state
- Provides consistency and high availability

**Event**
- Record of cluster activities
- Generated by Kubernetes components
- Useful for debugging and monitoring

**Eviction**
- Process of removing pods from nodes
- Triggered by resource pressure or node maintenance
- Respects PodDisruptionBudgets when possible

## F

**Finalizer**
- Key that prevents object deletion
- Allows cleanup of dependent resources
- Removed when cleanup is complete

**Flannel**
- Simple overlay network for Kubernetes
- Provides pod-to-pod connectivity
- Popular CNI plugin choice

## G

**Garbage Collection**
- Automatic cleanup of unused objects
- Removes objects when owner references are deleted
- Prevents resource leaks

**Grace Period**
- Time allowed for graceful pod termination
- Default is 30 seconds
- Can be customized per pod

## H

**Health Check**
- Mechanism to verify container health
- Types: Liveness, Readiness, Startup probes
- Used for automatic recovery and traffic routing

**Helm**
- Package manager for Kubernetes
- Uses Charts to define applications
- Manages releases and upgrades

**HorizontalPodAutoscaler (HPA)**
- Automatically scales pod replicas
- Based on CPU, memory, or custom metrics
- Maintains target utilization levels

## I

**Image**
- Read-only template for containers
- Contains application and dependencies
- Stored in container registries

**Image Pull Policy**
- Controls when images are downloaded
- Always, IfNotPresent, Never
- Affects startup time and bandwidth usage

**Ingress**
- API object for HTTP/HTTPS load balancing
- Routes external traffic to services
- Provides SSL termination and path routing

**Ingress Controller**
- Component that implements Ingress rules
- Examples: nginx-ingress, traefik, istio
- Runs as pods in the cluster

**Init Container**
- Container that runs before app containers
- Used for setup tasks and dependencies
- Must complete successfully before app starts

## J

**Job**
- Runs pods to completion
- Ensures specified number of successful runs
- Used for batch processing and one-time tasks

**JSON Path**
- Query language for JSON data
- Used in kubectl for output formatting
- Example: `{.metadata.name}` gets object name

## K

**kube-apiserver**
- Main API server component
- Validates and processes API requests
- Stores data in etcd

**kube-controller-manager**
- Runs core control loops
- Includes Deployment, ReplicaSet, Service controllers
- Ensures desired state matches actual state

**kube-proxy**
- Network proxy on each node
- Implements Service networking
- Manages iptables/ipvs rules for load balancing

**kube-scheduler**
- Assigns pods to nodes
- Considers resources, constraints, and policies
- Pluggable and customizable

**kubectl**
- Command-line tool for Kubernetes
- Communicates with API server
- Primary interface for cluster management

**kubelet**
- Node agent that manages pods
- Communicates with API server
- Manages container lifecycle on nodes

**Kubernetes**
- Container orchestration platform
- Automates deployment, scaling, and management
- Originally developed by Google

## L

**Label**
- Key-value pair attached to objects
- Used for organization and selection
- Example: `app=nginx`, `version=v1.2`

**Label Selector**
- Query that matches objects by labels
- Used by Services, Deployments, etc.
- Types: equality-based, set-based

**Liveness Probe**
- Health check that restarts unhealthy containers
- Prevents stuck or deadlocked containers
- Can be HTTP, TCP, or command-based

**LoadBalancer**
- Service type that provisions external load balancer
- Provides external IP for cluster access
- Requires cloud provider integration

## M

**Manifest**
- YAML or JSON file describing Kubernetes objects
- Contains apiVersion, kind, metadata, spec
- Applied using `kubectl apply`

**Master Node**
- Node running control plane components
- Manages cluster state and scheduling
- Also called Control Plane Node

**Metadata**
- Information about Kubernetes objects
- Includes name, namespace, labels, annotations
- Used for identification and organization

**Metrics Server**
- Cluster-wide resource usage collector
- Provides CPU and memory metrics
- Required for HPA and kubectl top

**Minikube**
- Tool for running Kubernetes locally
- Single-node cluster for development
- Supports multiple container runtimes

**Multi-tenancy**
- Sharing cluster among multiple users/teams
- Achieved through namespaces and RBAC
- Provides isolation and resource quotas

## N

**Namespace**
- Virtual cluster within physical cluster
- Provides scope for names and resources
- Enables multi-tenancy and organization

**Network Policy**
- Firewall rules for pod communication
- Controls ingress and egress traffic
- Requires CNI plugin support

**Node**
- Worker machine in Kubernetes cluster
- Runs pods and cluster services
- Can be physical or virtual machine

**NodePort**
- Service type that exposes port on all nodes
- Allows external access to cluster services
- Port range: 30000-32767

## O

**Operator**
- Custom controller for specific applications
- Encodes operational knowledge
- Extends Kubernetes with domain-specific logic

**Orchestration**
- Automated management of containerized applications
- Includes deployment, scaling, networking, storage
- Core function of Kubernetes

## P

**Persistent Volume (PV)**
- Cluster-wide storage resource
- Independent of pod lifecycle
- Provisioned by admin or dynamically

**Persistent Volume Claim (PVC)**
- Request for storage by user/pod
- Binds to available Persistent Volumes
- Specifies size, access mode, storage class

**Pod**
- Smallest deployable unit in Kubernetes
- Contains one or more containers
- Shares network and storage

**Pod Disruption Budget (PDB)**
- Limits voluntary pod disruptions
- Ensures minimum availability during updates
- Protects against simultaneous pod removal

**Pod Security Policy (PSP)**
- Cluster-level resource controlling security contexts
- Deprecated in favor of Pod Security Standards
- Controls privileged access and capabilities

**Probe**
- Health check mechanism for containers
- Types: Liveness, Readiness, Startup
- Can be HTTP, TCP, or exec-based

## Q

**Quality of Service (QoS)**
- Pod classification based on resource specs
- Classes: Guaranteed, Burstable, BestEffort
- Affects scheduling and eviction priority

**Quota**
- Resource limits for namespaces
- Controls CPU, memory, storage, object counts
- Prevents resource exhaustion

## R

**RBAC (Role-Based Access Control)**
- Authorization mechanism for API access
- Uses Roles, RoleBindings, ClusterRoles
- Controls who can do what in cluster

**Readiness Probe**
- Health check for service traffic routing
- Removes unhealthy pods from service endpoints
- Prevents traffic to non-ready containers

**Registry**
- Storage system for container images
- Examples: Docker Hub, ECR, GCR
- Can be public or private

**ReplicaSet**
- Ensures specified number of pod replicas
- Replaces failed pods automatically
- Usually managed by Deployments

**Resource**
- API object type in Kubernetes
- Examples: Pod, Service, Deployment
- Defined by apiVersion and kind

**Resource Quota**
- Limits resource consumption in namespace
- Controls CPU, memory, storage, object counts
- Enforced by admission controllers

**Rolling Update**
- Deployment strategy for zero-downtime updates
- Gradually replaces old pods with new ones
- Default strategy for Deployments

**Rollback**
- Reverting to previous version of application
- Supported by Deployments
- Uses revision history

## S

**Secret**
- Object for storing sensitive data
- Base64 encoded (not encrypted)
- Can be mounted as files or env vars

**Selector**
- Mechanism for grouping objects by labels
- Used by Services, Deployments, etc.
- Enables loose coupling between components

**Service**
- Stable network endpoint for pods
- Provides load balancing and service discovery
- Types: ClusterIP, NodePort, LoadBalancer

**Service Account**
- Identity for processes running in pods
- Used for pod-to-API authentication
- Each namespace has default service account

**Service Mesh**
- Infrastructure layer for service communication
- Provides security, observability, traffic management
- Examples: Istio, Linkerd, Consul Connect

**Spec**
- Desired state specification for objects
- Describes what you want resource to look like
- Processed by controllers

**StatefulSet**
- Manages stateful applications
- Provides stable network identities
- Ordered deployment and scaling

**Status**
- Current state of Kubernetes objects
- Maintained by controllers
- Shows actual vs desired state

**Storage Class**
- Defines types of storage available
- Enables dynamic volume provisioning
- Examples: fast-ssd, replicated-storage

## T

**Taint**
- Property of nodes that repels pods
- Used to dedicate nodes for specific workloads
- Requires matching tolerations on pods

**Toleration**
- Allows pods to be scheduled on tainted nodes
- Matches taints on nodes
- Enables specialized node usage

**Traffic Policy**
- Controls how traffic is routed to pods
- Local vs Cluster traffic policies
- Affects load balancing behavior

## V

**Volume**
- Directory accessible to containers in pod
- Persists data beyond container lifecycle
- Types: emptyDir, hostPath, persistentVolumeClaim

**VolumeMount**
- Specification for mounting volumes in containers
- Defines mount path and volume reference
- Can be read-only or read-write

## W

**Workload**
- Application running on Kubernetes
- Managed by controllers like Deployments
- Can be stateless or stateful

**Worker Node**
- Node that runs application workloads
- Contains kubelet, kube-proxy, container runtime
- Receives instructions from control plane

## Y

**YAML**
- Human-readable data serialization format
- Primary format for Kubernetes manifests
- Alternative to JSON for configuration

## Quick Reference Cards

### Resource Types
| Type | Abbreviation | Purpose |
|------|-------------|---------|
| Pod | po | Run containers |
| Service | svc | Network endpoint |
| Deployment | deploy | Manage pods |
| ConfigMap | cm | Configuration |
| Secret | secret | Sensitive data |
| Namespace | ns | Isolation |
| Node | no | Cluster machine |

### Common Commands
| Command | Purpose |
|---------|---------|
| `kubectl get` | List resources |
| `kubectl describe` | Detailed info |
| `kubectl apply` | Create/update |
| `kubectl delete` | Remove resources |
| `kubectl logs` | View logs |
| `kubectl exec` | Execute commands |

### Status Values
| Status | Meaning |
|--------|---------|
| Pending | Being scheduled |
| Running | Active and healthy |
| Succeeded | Completed successfully |
| Failed | Terminated with error |
| Unknown | Status cannot be determined |
